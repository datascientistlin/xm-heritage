# 开发规范

## 代码风格

### JavaScript
- 使用ES6+语法
- 使用驼峰命名法
- 使用4个空格缩进
- 文件末尾添加换行符

### 注释规范
- 函数注释：JSDoc格式
- 行内注释：简洁明了
- 重要逻辑需添加注释说明

### 命名规范
- 变量和函数：camelCase
- 常量：UPPER_SNAKE_CASE
- 类名：PascalCase
- 文件名：kebab-case

## 目录结构约定

### 后端
```
/backend/
├── /config/         # 配置文件
├── /controllers/    # 控制器 - 业务逻辑编排
├── /middleware/     # 中间件 - 请求处理逻辑
├── /models/         # 模型 - 数据结构定义
├── /routes/         # 路由 - URL映射
└── /services/       # 服务 - 核心业务逻辑
```

### 前端
```
/frontend/
├── /assets/         # 静态资源
│   ├── /images/     # 图片资源
│   └── /audio/      # 音频资源
├── /css/            # 样式文件
├── /js/             # JavaScript文件
└── index.html       # 主页面
```

## 安全规范

### 儿童安全
- 所有AI响应必须经过安全过滤
- 限制不适宜儿童的内容
- 响应长度不超过设定阈值
- 使用合适的系统提示

### 输入验证
- 所有用户输入必须验证
- 防止注入攻击
- 限制请求频率

### API安全
- API密钥不得硬编码
- 使用HTTPS连接
- 验证API响应

## 错误处理

### 统一错误格式
```javascript
{
  error: 'Error message',
  details?: 'Additional details',
  timestamp: 'ISO date string'
}
```

### 错误日志
- 记录错误级别和上下文
- 避免记录敏感信息
- 记录用户友好消息

## 测试规范

### 单元测试
- 业务逻辑必须有单元测试
- 测试覆盖率应达到80%以上
- 边界条件和异常情况需要测试

### 集成测试
- 验证各模块间的协作
- 测试API端点
- 测试错误处理流程

## 提交规范

### Git提交信息
```
type(scope): subject

body

footer
```

类型包括:
- feat: 新功能
- fix: 修复bug
- docs: 文档更新
- style: 代码格式调整
- refactor: 代码重构
- test: 测试相关
- chore: 构建过程或辅助工具变动

### 分支命名
- feature/feature-name
- bugfix/issue-description
- hotfix/critical-fix

## 部署规范

### 环境配置
- 生产环境使用production配置
- 开发环境使用development配置
- 环境变量不得提交到仓库

### 版本管理
- 使用语义化版本号
- 主要版本变化需要更新文档
- 向后兼容性保持

## 性能优化

### 前端优化
- 图片压缩和懒加载
- 代码分割和懒加载
- 缓存策略

### 后端优化
- API响应缓存
- 数据库查询优化
- 连接池管理

## CI/CD最佳实践

- 自动化测试通过后才能合并
- 代码质量检查
- 安全漏洞扫描
- 自动化部署

## 儿童安全特殊要求

- 所有AI生成内容必须经过安全过滤
- 响应长度和主题限制
- 不保存儿童个人数据
- 家长控制功能
- 内容举报机制
